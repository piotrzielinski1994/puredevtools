# Architectural Decisions - puredevtools

Append-only log of architectural and design decisions made during development.

## Format

Each entry follows this structure:

| Date | Decision | Rationale |
|------|----------|-----------|
| {YYYY-MM-DD} | {What was decided} | {Why this choice was made} |

## Decisions

| Date | Decision | Rationale |
|------|----------|-----------|
| 2026-07-09 | Cross-browser MV3 extension (Chromium + Firefox) from one codebase, built by Vite + `@crxjs/vite-plugin` | Core platform choice, expensive to swap. `@crxjs` handles MV3 bundling + HMR for both engines; alternative (webpack + hand-rolled manifest) rejected as more boilerplate |
| 2026-07-09 | Single typed manifest source (`buildManifest(target)` in `src/manifest/index.ts`) emits both Chrome (service_worker) and Firefox (scripts + gecko) variants; `TARGET` env selects the build, unknown value throws | Keeps the per-engine divergence (background shape, permissions, gecko settings) in one testable place instead of two hand-maintained `manifest.json` files that drift. Fail-fast on unknown target prevents silently shipping the wrong manifest |
| 2026-07-09 | Extension API via `webextension-polyfill` (`import browser`), never raw `chrome.*` | Promise-based, normalizes the Chromium/Firefox namespace + callback-vs-promise gap; one API surface across the codebase |
| 2026-07-09 | Vitest as the sole test layer for the scaffold (no Playwright/E2E yet) | Pure logic + React components cover the scaffold; a real-browser E2E layer is deferred until there is interception to drive |
| 2026-07-10 | Options page = master-detail shell (sidebar rule list + right tab strip/editor), replacing the single-column editor-swaps-list layout | Keeps the full rule list visible while editing; mirrors the requi workspace. Tab state isolated in a pure `useOpenTabs` hook (testable, no persistence in v1). pz-ddd/pz-archetypes evaluated - neither applies (pure presentation, no domain model) |
| 2026-07-10 | Single-mechanism response override (page layer only): deleted the whole network layer (`ChromeEngine`/`declarativeNetRequest`, `FirefoxEngine`/`webRequest`+`filterResponseData`, `RequestEngine` abstraction, `selectEngine`, `dataUrl`) plus the request-side/mock/status/latency actions and the `capabilities`/`diagnostics` plumbing. A rule now only forwards the real request and overrides response headers/body via the patched `fetch`/`XHR`. | The network layer was the sole source of Chrome/Firefox asymmetry (no Chrome body rewrite, no custom mock status/headers, Firefox-only latency). The page layer already forwarded+overrode identically on both engines, so the abstraction earned nothing. Cost: main-frame document navigation is no longer interceptable (documented limitation), matching peer tools. pz-ddd/pz-archetypes evaluated - neither applies (tooling reduction, no domain model) |
| 2026-07-12 | Rules are stored as a **workspace tree** (`FolderNode \| RuleNode`, arbitrary nesting, mixed root) instead of a flat `Rule[]`. `RuleRepository.getAll()` returns a **DFS pre-order `flatten`** of the tree, so the engine keeps consuming an ordered `Rule[]`; `Rule.priority` is removed and tree position is the sole ordering. Match precedence = visible top-to-bottom tree order. | Folders + hand drag-reorder are a UI/storage concern only: the engine already treated the list as an ordered first-match-wins sequence (`decide.find()` never read `priority` numerically), so flattening the tree leaves `engine/**`, `content/bridge.ts`, `match.ts`, `decide.ts` untouched. Alternative (cosmetic folders keeping flat priority) rejected - "what you see = what matches" is the intuitive contract. Pre-release, so the stored + portable format changed without a version bump. pz-ddd/pz-archetypes evaluated - neither applies (UI/storage reorg of existing model, ported requi's proven nested-tree structure). |
| 2026-07-12 | Deleting a folder deletes its **entire subtree** (contained rules + subfolders) after a `window.confirm`, rather than reparenting the contents up. | Matches the requi reference behavior 1:1 and keeps the delete mental model simple (the folder IS its contents). Alternative (move children to parent on delete) rejected for parity + simplicity; the confirm dialog guards the destructive case. |
| 2026-07-19 | Reintroduce **request-side mutation** (partial reversal of the 2026-07-10 response-only decision): two new `RuleAction` variants `modifyRequestHeaders` (set/remove `HeaderOp[]`) + `rewriteRequestBody` (string). The page layer now mutates the outgoing request (header ops + optional body replacement) *before* forwarding, in addition to the existing response override. Exposed via a third "Request" editor tab mirroring the Response tab. `Interception` override gained flat `requestHeaderOps`/`requestBody` beside the response `headerOps`/`body`; request content-type is set via a header row (no dedicated field); GET/HEAD + body is applied as-is; XHR `remove` is a documented no-op. | The 2026-07-10 deletion removed request-side actions along with the network layer because that layer was the source of Chrome/Firefox asymmetry - but request mutation itself works identically on both engines through the same `fetch`/`XHR` patch, so it carries no per-engine cost. Users need to change what the page *sends* (auth/env headers, payload tweaks), the core job of ModHeader/Requestly-class tools. Kept the single page-layer mechanism (no network layer reintroduced) and the flat `Interception` shape to minimize churn. pz-ddd/pz-archetypes evaluated - neither applies (dev tooling, no domain model); pz-codebase-design not deeply invoked - the change mirrors the established Response-side interface 1:1, no new seam. |
| 2026-07-18 | Persist the options-page open-tabs working set (`openKeys` + `activeKey`) to a dedicated `browser.storage.local` key (`puredevtools.openTabs`), reversing the 2026-07-10 "no persistence in v1" deferral. Injected a narrow `TabsStore` port (`load`/`save`) into `useOpenTabs` (opts arg) with the browser + zod impl in a coverage-excluded `createTabsStore.ts`; the reducer stays pure. Draft (`new:draft`) is never persisted; hydration is gated on rules-`ready`; deleted-rule tabs are filtered on load; malformed/missing storage â†’ empty state. No cross-window live sync (last-write-wins). | Reopening the options page lost the working set. The seam keeps the pure reducer + its 15 tests intact and isolates I/O exactly as `useTheme`/`createGateway` do. Gating on `ready` avoids pruning against an empty rule set mid-load. Cross-window `storage.onChanged` sync rejected as YAGNI (tabs are a per-window working set). pz-ddd/pz-archetypes evaluated - neither applies (presentation state, no domain model); pz-codebase-design invoked for the port shape. |
| 2026-07-19 | Add **pre/post scripts**: two new `RuleAction` variants `preScript`/`postScript` (each a `source: string`), run against the request/response by the page layer. Scripts execute via **`AsyncFunction` in the page MAIN world** (not a QuickJS-WASM sandbox), exposing a scoped `req`/`res`/`console` facade (no variable/env system, no `getVar`/`setVar`). A re-entrancy guard passes through any `fetch`/`XHR` issued while a script runs. Errors + CSP-blocked construction are logged and the script is skipped (declarative ops still apply); no execution timeout. New deep module `src/engine/page/script/` (`runScript` + `facades`) hides the `AsyncFunction`, guard, and mutable req/res model; the editor is CodeMirror (`@uiw/react-codemirror`) + in-browser ESLint (`eslint-linter-browserify`, `no-undef`, stage-aware globals) + stage-aware autocomplete, ported from `purerequest` into `src/ui/shared/script/` + `ScriptEditor.tsx`; exposed via a fourth "Scripts" editor tab (Pre/Post). | Declarative header/body ops cannot express conditional logic, computed values, or partial JSON reshaping - the standard power-user tier. Chose `AsyncFunction` over QuickJS-WASM because `purerequest` can only use QuickJS thanks to Tauri's `csp:null`; injected into arbitrary pages a WASM sandbox needs the page's `wasm-unsafe-eval` and degrades on the same strict-CSP sites as `AsyncFunction` but with far more weight, so it earns nothing here. Scripts are the user's own trusted code. Scoped `req`/`res` API (no vars/env) keeps this single-ticket-sized; a variable subsystem is deferred (YAGNI). Kept the single page-layer mechanism + flat `Interception` shape (`preScript?`/`postScript?`), mirroring the 2026-07-19 Request-tab precedent. pz-ddd/pz-archetypes evaluated - neither applies (dev tooling, no domain model); pz-codebase-design invoked for the `src/engine/page/script/` module seam. First code-execution feature - a deliberate departure from the "all rule behavior is declarative data" invariant, bounded by the CSP-degrade + re-entrancy guard. |
| 2026-07-19 | Replace `eslint-linter-browserify` in the Scripts-tab `no-undef` linter with a hand-rolled scope check over **`espree` (parse) + `eslint-scope` (scope analysis) + `globals` (builtin set)**. `jsUndefLinter` (`src/ui/shared/script/script-eslint.ts`) now parses the source, walks the scope manager's global `through` references, and flags any not in `globals.builtin` âˆª the stage globals (`req`/`console`, plus `res` for post). Same behavior, same 9 ACs, zero eval. | The `eslint-linter-browserify` `Linter` loads rule metadata through **`ajv`, which compiles each rule's options JSON-schema with `new Function`** - a string-eval. The options page is an extension page under MV3 CSP `script-src 'self'` (no `unsafe-eval`), so opening the Scripts tab threw *"Evaluating a string as JavaScript violates the following Content Security Policy directive"* and the linter crashed. It passed vitest+build because jsdom/Node CSP is unrestricted - the constraint is invisible outside a loaded extension. Ported blindly from `purerequest` (a Tauri app with `csp:null`, so eval is fine there). `espree`+`eslint-scope`+`globals` are acorn-based, do no eval, and are exactly `no-undef`'s internals minus the ajv rule-schema layer. Relaxing the extension CSP to allow `unsafe-eval` was rejected outright (security regression, and MV3 forbids it for extension pages). |
| 2026-07-20 | Add **cookie sync** as a **separate subsystem** (`src/cookies/` + `src/ui/cookies/`), NOT a rule: copy named cookies from a source URL to a target URL on manual "Sync now". Requires the first new manifest permission since scaffold - `"cookies"` added to `permissions` for both engines (`host_permissions` already `<all_urls>`). Pure core `syncMapping(mapping, port)` takes an injected `{getAll,set}` cookie-API port; it reads `cookies.getAll({url:source})`, filters to an explicit `cookieNames` allow-list, and writes each via `cookies.set({url:target,...})` **omitting `domain`** (so the target host from `url` governs scope - copying the source `.prod.com` domain would make `set` reject a `localhost` target) and **dropping `secure` when the target scheme is `http:`** (a `Secure` cookie cannot be written to `http://localhost`). Per-cookie set failures and absent names are collected into a `SyncResult.skipped` and never abort the batch. Stored under its own `browser.storage.local` key `puredevtools.cookieSync` (flat `CookieMapping[]`, own `.strict()` schema, NOT merged into the rules `portableSchema`); no entry ever touches the workspace tree / `flatten` / `decideInterception`. Runs in the options page (privileged extension page - direct `browser.cookies` access, no background relay). Surfaced via a top-level `Rules \| Cookie sync` view switch in options. | The page-layer `fetch`/`XHR` patch (the single interception mechanism) fundamentally cannot do this: it cannot read HttpOnly cookies (`document.cookie` hides them) nor set the forbidden `Cookie` request header, and auth cookies are almost always HttpOnly + server-set. Only the `cookies` API reads/writes HttpOnly cookies cross-domain, so a new permission is unavoidable - the core "prod session on localhost" job of Requestly/EditThisCookie-class tools. Kept it fully orthogonal to the rules model (own store + schema + UI section) rather than shoehorning a cookie mapping into the `Rule`/tree model, which would break the tree-agnostic flat-ordered `Rule[]` engine contract. Manual-only trigger + name allow-list + drop-secure-on-http + separate options section were the four product decisions (auto-sync via `cookies.onChanged`, copy-all-by-domain, and import/export UI deferred as YAGNI; `enabled` field reserved for future auto-sync). No background code because the relay path exists only for interception reports. Caveat documented (README, not code): a copied token can still be rejected server-side if bound to origin/CSRF/UA - convenience tool, not a session-transfer guarantee. Cookie **values** are never logged (names + counts only). pz-ddd/pz-archetypes evaluated - neither applies (dev tooling, no domain model); pz-codebase-design invoked for the `CookieApiPort` seam (mirrors the `TabsStore`/`decide`-resolves-then-patch-layers-dumb precedent). |
| 2026-07-20 | Add **request URL rewrite**: a new declarative `RuleAction` variant `{ type: 'rewriteRequestUrl'; target: string }` that redirects a matched request to another URL before forwarding (prod -> localhost/host swap). Semantics **auto-detected from the target** by a pure `resolveRewrite(originalUrl, target)` (`src/engine/page/rewriteUrl.ts`): an origin-only target (parsed `pathname==='/' && !search && !hash`, trailing slash allowed) swaps scheme/host/port and keeps the original path/query/hash; a target with an explicit path replaces the URL but backfills the original query/hash the target omits; empty/unparseable target is a no-op. Resolution runs once **in `decideInterception`**, which puts the final absolute URL on the flat `Interception` override as `requestUrl?`; the patch layers just forward to it (`patchFetch`: `new Request(url, input)` for Request-object inputs to preserve method/headers/body; `patchXhr`: re-`open`s the delegate and re-applies the recorded request headers). The declarative rewrite runs **before** any pre-request script, so a script's `setUrl` still wins. Surfaced as an input on the existing Request tab (no new tab, no mode toggle). | Pointing matched traffic at a different backend is the core "map remote"/host-redirect job of Requestly/Charles-class tools; a pre-script `setUrl` already covered it imperatively but the declarative no-JS field is the common case. Auto-detect over a mode enum keeps it to one input field (YAGNI on a toggle). Resolving in `decideInterception` keeps the fetch/XHR patch layers dumb and trivially testable, mirroring how header/body ops are pre-resolved. Kept the single page-layer mechanism + flat `Interception` shape (`requestUrl?`), extending the 2026-07-19 Request-tab precedent 1:1. pz-ddd/pz-archetypes evaluated - neither applies (dev tooling, no domain model); pz-codebase-design - only new seam is the pure `resolveRewrite`, mirrors the established Request-side interface. |
| 2026-07-20 | Add a **full keyboard-shortcut subsystem** ported from `purerequest`: central action registry (`src/shortcuts/registry.ts`) + `resolveShortcuts`/`findConflict` + single `useActionHotkeys(handlers)` hook over `@tanstack/react-hotkeys`, `event.code`-aware recorder, per-action **rebinding** settings UI (third options view), and full **roving-tabindex** sidebar-tree keyboard navigation (`tree-keyboard.ts` resolver -> `tree-nav.tsx` seam -> `TreeRow`). Overrides persist under a new `browser.storage.local` key `puredevtools.shortcuts` via `ShortcutsProvider`, live-synced across all three roots (options/popup/devtools) by `storage.onChanged`. **In-page listeners only - no manifest `commands`** (no new permission). Wired on all three surfaces; generic ids `new-item`/`delete-item` are **contextual** (each mounted options view supplies its own handler). Removed the stale placeholder `Mod+S` "Saved" toast from cookie sync; `Mod+Enter` now syncs the selected mapping for real. | The app had only scattered inline `useHotkeys` calls (one real `Mod+S`, one fake cookie-sync placeholder). A first-class rebindable system is the standard power-user tier and the settings list doubles as discoverability (no separate cheatsheet/palette - deferred). Three decisions were forced, not stylistic: (1) **default keys diverge from purerequest** - it's a Tauri desktop app where `Cmd/Ctrl+T/W/N/Tab/Shift+I` are free, but puredevtools runs in browser tabs/popup/devtools where the browser reserves them and a page `keydown` cannot `preventDefault` them; the mechanism ports verbatim, the `defaultHotkey` strings do not, and `registry.test.ts` guards defaults against a reserved-combo blocklist. (2) **`commands` manifest API rejected** (user decision) - keeps zero new permissions and mirrors purerequest 1:1; a global open-popup key can be added later behind an ADR. (3) **contextual generic ids over per-view ids** - avoids id/key duplication + false conflict flags, and needs no cross-component reaching because only one view mounts at a time. Rebinding settings-view chosen as the discoverability surface (command palette deferred as YAGNI, per user). pz-ddd/pz-archetypes evaluated - neither applies (UI/tooling, no domain model); pz-codebase-design applied - faithful port of purerequest's already-deep registry/resolve/hook modules, with the pure `eventToHotkey` split from the `useRecordHotkey` hook across node/jsdom test envs and the shortcuts context split from its storage-backed provider so polyfill-free consumers (SidebarTree) don't eval `webextension-polyfill` at import. |
